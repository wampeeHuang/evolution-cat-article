# 信息密度门禁

> Step 3 执行。骨架层，不可通过 retro 回流修改。

---

## 一、开头硬门禁（Step 3 落笔前）

**PASS 条件**：
- 第一句是具体事件/数据/场景，不是抽象判断
- 午休刷手机的人读到第三段，能说出这篇文章要讲什么
- **开头制造悬念但不揭示核心意外。检验：删掉正文只读开头部分，读者能说出全文结论 → FAIL。** 钩子引发"为什么？""怎么做到的？"，不给出解析框架。翻牌点应在 outline-review.md 指定的位置（通常 Section 03）

**禁止**：
- "在当今…时代""随着…发展""近年来…"
- 前三个自然段出现 ≥3 个不认识的专有名词

不通过 → 重写开头，不继续。

## 二、认知引渡

文章开头每一句锚定读者已有认知，从已知渡到未知。

1. 第一段锚定读者大概率刷到过的事件/产品/人名
2. 新人物从小红书/GitHub/X 的主场切入，不扔进陌生圈子
3. 专有名词首次出现：类比在前，术语在后（括号）
4. 抽象概念：先给日常画面，再解释概念

## 三、风格硬指标

| # | 指标 | PASS 标准 |
|---|------|----------|
| 1 | 字数 | 2000-6000 |
| 2 | 信息密度 | 每 300 字 ≥1 个可验证的数字或专有名词 |
| 3 | AI 高频词 | 命中 = 0 |
| 4 | 最长句 | ≤ 45 字 |
| 5 | 最短句 | 可 2-3 字，用于节奏变化 |
| 6 | 段落 | 自然段，不每句换行 |
| 7 | 术语首次出现 | 类比在前，术语在后 |
| 8 | 不确定的事 | "我也没完全搞懂"，不糊弄 |
| 9 | 主观表达 | "说实话""我觉得"每段最多 1 次 |
| 10 | 结尾 | `这里是进化猫，陪你一起看世界✨` |

## 四、术语降维

读者的注意力是货币。每遇到一个不懂的词花掉一枚。最多花到第三枚他就划走了。

| 术语 | ❌ 不降维 | ✅ 降维后 |
|------|----------|----------|
| MCP Server | 通过 MCP Server 接入 AI Agent | 你可以理解成一个"翻译官"——AI 说话，网页说话，它坐中间把两边翻译通（技术圈叫 MCP Server） |
| Playwright | Playwright 容器处理 JS 渲染 | 一个机器人浏览器——你给它网址，它能帮你点按钮、等加载（这叫 Playwright） |
| Readability | 基于 Mozilla Readability 算法 | Firefox 浏览器"阅读模式"用的那个提取技术（官方名叫 Readability） |

## 五、禁止词库

### 🔴 绝对禁止

| 表达 | 替代思路 |
|------|---------|
| "标志着""代表了""证明了" | 直接说事实 |
| "此外""至关重要""值得注意的是" | 删除，直接进入下一段 |
| "不仅……而且……" | 断成两句 |
| "首先其次最后""从以下几个维度" | 用视觉结构代替 |
| "这意味着什么""这对你意味着什么""这对你有什么用""这对你有什么启示" | 删掉 rhetorical question，用具体锚点替代 |
| "随着 AI 技术的发展……"（万能开头） | 第一句直接甩结论或数据 |
| 三段式总结结尾 | 结尾是转发钩子，不是总结 |
| "希望这对您有帮助" | 删除 |

### 🟡 谨慎使用

| 表达 | 何时可用 | 何时不可用 |
|------|---------|-----------|
| "推荐" | 亲自用过且标注了适用条件 | 二手研究时不能说推荐 |
| "最佳实践" | 有具体场景限定 | 泛泛而谈 |
| "深度好文""干货满满" | 不可用——让读者判断，不自夸 | — |
