# 进化猫 · 文章写作与发布

> 把飞书「已入选」选题变成微信草稿箱里的可预览文章。
> 这不是查阅文档，这是执行。读取后立即开始，不等待确认。

## Phase 0：初始化检查（首次运行）

检查 skill 目录下是否存在 `_runtime/paths.json`：

**不存在 → 首次运行**，逐项询问用户：
1. "config.yaml 的完整路径？"（飞书 + image API 配置）
2. "文章输出到哪个目录？"（默认 `./output`）
3. "wechat-publisher skill 是否已安装？"

将答案写入 `_runtime/paths.json`：
```json
{"config_path": "用户回答", "output_dir": "用户回答", "wechat_publisher": true}
```

**存在 → 直接读取**，跳过询问。

---

## 前置条件

执行前从飞书拉取状态=`已入选` 的选题，获取字段：中文标题、链接、来源、日报概览、入选理由。

无已入选选题 → 终止，回复"飞书表格中无已入选选题，请先跑选题SOP。"

## 路径约定

本 skill 内所有路径以 Phase 0 缓存的 `_runtime/paths.json` 为准：

| 变量 | 来源 | 用途 |
|------|------|------|
| `{config_path}` | paths.json → config_path | config.yaml 完整路径 |
| `{output_dir}` | paths.json → output_dir | 文章 md + 配图输出目录 |
| `references/perspectives/` | skill 内置 | 4 个写作视角 skill（mrbeast/tim/karpathy/humanizer-zh） |
| wechat-publisher | 外部 skill | 发布到微信草稿箱（Step 7） |

读取 `{config_path}` 获取：飞书配置（app_id/app_secret/base_token/table_id）、image API 配置（api_url/api_key/model）。

## 两层架构

| 层 | 内容 | 策略 |
|----|------|------|
| **铁律层** | 事实校验、信息密度、叙述诚实性、去AI味、阶段门禁、必落盘 | **不可跳过** |
| **节奏层** | 用什么结构、怎么开头、选什么节奏原型 | **每次由选题决定** |

SOP 管质量底线，不管文章形状。形状由话题本身决定。

## 读者恒定画像

| 维度 | 定义 |
|------|------|
| 身份 | 会使用AI产品的互联网人（产品/运营/设计/管理/学生），不是写代码的工程师 |
| 技术基线 | 知道 Token、API、开源大概是什么；不理解协议、框架、具体库名 |
| 阅读场景 | 微信公众号 + 手机竖屏。碎片时间（通勤/排队/午休），注意力高度分散 |
| 阅读动机 | 发现能立刻用的AI工具或方法，不想被AI浪潮抛下 |
| 弃读原因 | 连续两个自然段看不懂 → 划走；满屏技术名词没有解释 → 划走 |

**每句话自问**：一个午休时间刷手机看到这句话的人，能一眼看懂吗？

## 执行流程

### Step 1：全网搜索 — 原始素材采集

对每个入选选题执行 5 维度搜索：
1. 原始新闻来源（官方发布/博客/X）
2. 科技媒体解读（TechCrunch/36氪/虎嗅等）
3. 社区讨论（Reddit/HN/微博）
4. 行业数据/报告
5. 中国社区用语（小红书/即刻，提取读者熟悉的表达方式）

**事实校验铁律**：
- 禁止使用模型生成的虚假链接
- 禁止保留无法访问的死链
- 禁止标注"待确认"后照常发布
- 无法验证的内容必须删除，不留、不标、不凑数

**素材包输出**：摘要 / 关键数字 / 各方反应 / 可信度评估（新闻解读/观点论证型适用）/ 适用论点（同上）

**门禁**：≥3 个独立来源印证核心事实 → 进入 Step 2。不通过则扩大搜索或放弃该选题。

---

### Step 2：创意会 — 三道串行门禁

三道门禁**串行执行**，后者不改前者的决定。每道门只回答"过"或"不过"。

| 顺序 | 门禁 | 负责人 | 做什么 | 不通过 |
|------|------|--------|--------|--------|
| 第一道 | 标题关 | MrBeast | 极端对比/数字锚定/赌注型标题，让午休刷手机的人忍不住点进来 | 重写，不推进 |
| 第二道 | 壳核关 | Tim | 标题↔封面↔内核互译检查：只看标题能脑补封面吗？只看封面能猜到文章要说什么吗？ | 对齐后重过 |
| 第三道 | 诚实关 | Karpathy | 有没有夸大/吹牛/编数据？标题承诺的事文章里兑现了吗？ | 修正夸大措辞 |

执行方式：读取 `references/perspectives/mrbeast-perspective.md`、`tim-pan-perspective.md`、`karpathy-guidelines.md` 获取各角色的判断标准。

**强制选一个节奏原型**（全自动场景禁选故事叙述型）：

| 原型 | 适用场景 | 结构 |
|------|---------|------|
| 工具上手型 | 新产品/工具/方法 | 解决什么问题 → 别人实测效果（**必须引用出处**）→ 怎么复现 |
| 新闻解读型 | 热点事件/行业动态 | 发生了什么 → 为什么重要 → 对你意味着什么 |
| 观点论证型 | 反直觉判断/趋势预测 | 反直觉结论 → 证据 1/2/3 → 你的行动 |

**全自动场景叙述铁律**：全文一律使用引用式叙述，禁止伪造第一人称体验。
- ❌ "我试了一下……""我部署了……"
- ✅ "作者在 Reddit 上分享了 benchmark 数据——[具体数字]"
- ✅ "从社区反馈来看，[体验]是……"

**循环上限**：每道门禁最多 2 轮。2 轮后仍有争议 → 采纳 Karpathy 的最后判断（诚实关权重最高）。

**门禁**：三道全部通过 + 能用一句话说出"读者看完会改变什么想法或知道该做什么" → 进入 Step 3。

---

### Step 3：写文章

**写作人设**：趁午休在微信上给朋友安利刚发现的好东西。不装专家，装爱分享的朋友。

**风格硬指标（逐条检查）**：
- 字数：2000-6000
- 每 300 字至少 1 个可验证的具体数字或专有名词
- AI 高频词命中 = 0（列表见 Step 4.2）
- 最长句 ≤ 45 字，最短句可以只有 2-3 字
- 段落是自然段，不是独立句——完整想法在同一个段落里
- 术语首次出现：类比在前术语在后，类比用日常场景（快递分拣、搬家、停车场、菜谱）
- 全自动场景：全文引用式叙述，禁止伪造第一人称体验
- 文末：参考来源 + `这里是进化猫，陪你一起看世界✨`

**Section 标题格式**：数字编号 + 核心主题（如 `01 从"对话工具"到"操作电脑"`）

**门禁**：读一遍无不自然段落 + 2000-6000 字 + 引用可追溯到素材包 → 进入 Step 4。

---

### Step 4：打磨 — 编辑初审 + 去AI味

**4.1 AI 编辑初审**

逐段扫描三个维度：

| 维度 | 检查内容 | 不通过时 |
|------|----------|----------|
| 校对 | 错别字、病句、标点错误、重复字/词 | 修正 |
| 可读性 | 单句>45字、段落>200字、连续5段无数字或专有名词 | 拆句或补充 |
| 节奏 | 连续3段等长度无变化、连续3段无标点 | 调整 |

修具体问题，不全文重写。

**4.2 去 AI 味**

读取 `references/perspectives/humanizer-zh.md`，执行一轮 Humanizer-zh：
- 删除"标志着""代表了""证明了"
- 删除"此外""至关重要""值得注意的是""深入探讨"
- 打破"不仅…而且…"结构
- 变化句长
- 信任读者，不解释隐喻
- 删掉所有"希望这对您有帮助"类协作痕迹

**门禁**：初审三维度全部通过 + 全文 0 命中 AI 高频词 → 进入 Step 5。

---

### Step 5：生成配图

**Step 5.0：参考提示词库（不可跳过）**

生图前查 [awesome-gpt-image-2](https://github.com/freestylefly/awesome-gpt-image-2) 案例库（397 cases / 13 categories / 20+ templates）：

1. **匹配分类**：根据文章类型选 gallery 分类（工具→UI+产品，新闻→信息图+海报，观点→插画+摄影）
2. **查案例**：web_fetch 拉取 GitHub raw URL，找 2-3 个视觉方向匹配的案例
3. **提取模式**：提取提示词结构顺序、风格关键词栈、负面约束、比例约定
4. **构建提示词**：以提取模式为骨架填入文章内容

案例索引：
- 模板浏览：`https://gpt-image2.canghe.ai/#templates`（可视化浏览模板效果）
- 提示词源文件（GitHub raw）：
  - `https://raw.githubusercontent.com/freestylefly/awesome-gpt-image-2/main/docs/gallery-part-1.md`
  - `https://raw.githubusercontent.com/freestylefly/awesome-gpt-image-2/main/docs/gallery-part-2.md`
  - `https://raw.githubusercontent.com/freestylefly/awesome-gpt-image-2/main/docs/templates.md`

**Step 5.1：三级降级策略**

| 优先级 | 方案 | 说明 |
|--------|------|------|
| 1 | gpt-image-2 API | 从 config.yaml 读 image 段。POST body: `{"model":"gpt-image-2","prompt":"英文提示词","n":1,"size":"1792x1024"}`，全部 16:9 横版，timeout≥180s |
| 2 | 即梦网页版 | API 失败/超时后降级。浏览器自动化：打开 `jimeng.jianying.com/ai-tool/image/generate`，中文提示词 |
| 3 | 提示词 txt | 前两级都失败，生成 `image-prompts.txt` 供人工处理 |

**产出**：封面图 × 1（文件名以「封面」开头）+ 配图 ≥ 2 张，全部 16:9 横版，英文提示词。

**门禁**：封面图 1 张 + 配图 ≥ 2 张 → 进入 Step 6。优先级 3（txt）不视为通过，终审第 8 项标记"配图待补"。

---

### Step 6：AI 终审 — 11 项检查

| # | 检查项 | 不通过 |
|---|--------|--------|
| 1 | 数据复核：每个数字反向核对素材包 | 修正或删除 |
| 2 | 标题：冲击力 + 壳核对齐 + 诚实 | 回 Step 2 |
| 3 | 封面图：存在且与标题互译通过；仅 txt 时标记"待补"不阻断 | 回 Step 5 |
| 4 | 字数：2000-6000 | 回 Step 3 |
| 5 | 叙述诚实：全文引用式，无编造第一人称 | 修正引用 |
| 6 | 术语降维：类比在前术语在后 | 补类比 |
| 7 | AI味：Humanizer-zh 跑过，高频词 0 | 回 Step 4 |
| 8 | 配图：≥2张且绑定具体段落；仅 txt 时标记"待补" | 补充 |
| 9 | 参考来源：逐条打开URL确认可访问 | 删除死链 |
| 10 | 结尾：`这里是进化猫，陪你一起看世界✨` | 补上 |
| 11 | 落盘：md + 封面图 + 配图均已保存到 `{output_dir}/YYYYMMDD_标题/` | 确认文件 |

**迭代上限**：终审最多跑 2 轮。2 轮后仍有不达标项 → 标注在文章日志中，交付当前最优版本，不无限打磨。

**门禁**：11 项中除配图"待补"标记外全部通过 → 进入 Step 7。

---

### Step 7：发布到微信草稿箱

发布由 wechat-publisher skill 执行。若 Phase 0 确认 wechat-publisher 已安装，直接调用其文章发布流程。

**发布参数**：theme=evolution-cat, color=blue, author=黄皮黄, publish_method=browser。

传入 md 文件路径给发布脚本，脚本内部处理格式转换。不手动预转 HTML。

失败时（Chrome 未登录/扫码过期）→ 通知用户手动处理，md + 配图已落盘不丢失。

## 诚实边界

### 模型能力边界
- **事实准确性**：模型不具备实时网页浏览能力时，Step 1 搜索可能产出过时信息。所有引用以实际可访问 URL 为准，不以模型训练数据为准。
- **中文表达**：模型对中文互联网社区用语（小红书/即刻风格）的把握可能偏正式，Step 4 去AI味不能完全替代人工润色。
- **配图质量**：gpt-image-2 / 即梦对中文概念的表达可能不准确，复杂场景建议人工生图。

### 外部依赖故障
- **飞书 API 不可用**：前置条件拉取失败 → 终止流程，通知用户检查飞书配置。
- **gpt-image-2 API 超额/故障**：自动降级到即梦，再失败降到 txt。不影响文章正文产出。
- **Chrome 未登录微信**：Step 7 发布失败 → 通知用户手动扫码登录后重试，或改用手动发布。md + 配图已落盘不丢失。
- **wechat-publisher 脚本报错**：检查 bun 是否安装、node_modules 是否完整。

### 全自动场景限制
- 禁止使用「故事叙述型」节奏原型（没有亲身经历可用）。
- 全文引用式叙述 → 读者可能感觉"缺少一手体验"。如果选题必须一手体验才有说服力，放弃该选题。
- 网上反馈不够（没评测、没人讨论）→ 选题不值得写。

### 内容类型边界
- **适用**：AI工具上手、AI新闻解读、AI趋势观点。
- **不适用**：纯技术教程（需要代码逐行解释）、学术论文解读（需要领域专业知识）、访谈/人物特写（需要真实采访素材）。

### 信息时效
- 写作视角 skills（mrbeast/tim/karpathy/humanizer-zh）的方法论提取自 2025 年公开资料，未跟踪后续更新。
- awesome-gpt-image-2 案例库状态以 GitHub 仓库为准，案例数量可能变化。
- 本 skill 制定于 2026-05，微信公众号发布流程如有变化需更新 Step 7。

### 已知未覆盖
- 多平台分发（小红书/知乎/即刻）：同一 md 可复用为内容源，但本 skill 不包含微信以外的平台适配。
- 发布后数据追踪（阅读量/分享/评论）：不在本 skill 范围内。
